# 容器运行时事件能力设计

本文档定义 `Graft` 容器模块的 `Runtime Events` 能力设计，覆盖当前架构盘点、authority 决策、runtime-neutral
domain model、backend / frontend 实施路径、streaming 复用策略、provider extensibility 和 phased rollout。

本设计的目标是提供 runtime-agnostic 的容器事件能力，当前首个 provider 为 Docker，但上层产品、契约和 UI 不得把
Docker 当作唯一长期真相。

本文档是本主题的 design authority，不替代：

- `ai-plan/design/容器管理设计.md`
- `ai-plan/design/容器资源状态与订阅治理设计.md`
- `ai-plan/design/服务端API边界与兼容治理规范.md`
- `ai-plan/design/前端架构设计.md`
- `ai-plan/design/契约治理与魔法值治理规范.md`

## 1. 背景与目标

当前容器 troubleshooting 的理想路径应为：

```text
Overview
  -> Events
  -> Logs
  -> Terminal
```

用户在读日志前，通常更需要先看到容器生命周期与异常信号，例如：

- `Created`
- `Started`
- `Restarted`
- `Stopped`
- `Removed`
- `OOMKilled`
- `Health Status Changed`
- `Exec Started`
- `Exec Finished`

第一版底层 provider 仍然是 Docker，但 `Graft` 的产品 IA 是“容器管理”，不是 “Docker Events”。

本设计目标：

- 把容器事件能力定义为 container module-owned runtime feature
- 让 Docker 只作为第一个 provider adapter
- 复用当前统一 realtime transport，而不是新增第二套 websocket 基础设施
- 为首个交付面 `Container Detail -> Events` 提供 bounded history + live stream
- 为未来 Podman、containerd、Kubernetes 等 provider 扩展预留 source interface，而不是把 Docker 概念抬升到上层

非目标：

- 不在本轮引入持久化事件表
- 不把 shell 专用 websocket 改造成通用事件通道
- 不把 Kubernetes Pod 级事件直接塞进当前 container detail contract
- 不在 contract 中写 provider-authored `message`

## 2. 当前架构盘点

### 2.1 Container Module

容器模块 authority 位于：

- `server/modules/container/**`
- `web/src/modules/container/**`

当前 live capability 包括：

- 列表、详情、日志、Shell、stats realtime、dashboard summary realtime
- 统一 container module `service`
- runtime adapter boundary：`server/modules/container/runtime.go`

当前 `Runtime` 已承载：

- `List`
- `Detail`
- `Logs`
- `StreamLogs`
- `Shell`
- `Start / Stop / Restart / Remove`

当前 `StatsCollectorRuntime` 作为 `Runtime` 的 sibling extension，说明容器模块已经接受“基础 runtime 能力 + 可选扩展接口”
的治理方式。

### 2.2 Runtime / Provider Abstraction

当前 runtime 抽象仍由 container module own：

- `Runtime`
- `StatsCollectorRuntime`

第一版 live runtime implementation 是 Docker adapter，但上层产品命名、OpenAPI 和 frontend module 都已经按“容器管理”
收敛，而不是按 Docker product 命名收敛。

### 2.3 Realtime Infrastructure

当前统一 realtime transport 位于：

- `server/internal/realtime/subscription.go`
- `server/internal/realtime/http_handler.go`
- `server/internal/realtime/ws_handler.go`
- `server/internal/realtime/hub.go`

已有模式：

- `POST /realtime/subscriptions` 负责 topic ticket issuance
- `GET /ws` 负责统一 websocket gateway
- `TopicIssuerRegistry` 负责 topic prefix ownership
- `Hub` 负责 topic publish / subscribe

当前 Hub 特性：

- subscriber buffer 固定为 `8`
- publish 时采用 `drop oldest, keep latest` 策略
- 支持 topic active / inactive observer

### 2.4 Existing Realtime Features

现有容器模块 realtime topic：

- `container.stats.list`
- `container.stats:{id}`
- `container.logs:{id}`
- `container.dashboard.summary`

现有 realtime 模式分为两类：

1. Boot-started always-on publish
   - `statsCollector`
   - 适合“用户打开页面前也应该持续采集”的能力
2. topic-observer-driven lazy stream
   - `logTopicStreamer`
   - 适合“高成本、只在用户订阅时才值得启动”的能力

### 2.5 Web Observability / Stream State

前端当前可复用能力包括：

- `web/src/shared/realtime/ws-client.ts`
- `web/src/shared/observability/stream-viewport-state.ts`
- `web/src/shared/observability/log-ring-buffer.ts`
- `web/src/modules/container/shared/stats-manager.ts`

当前 shared observability 语义：

- websocket reconnect 已有统一策略
- visual stream state 已有统一 resolver
- ring buffer 实现虽然命名是 `LogRingBuffer<T>`，但底层已经是 generic buffer
- container module 已经有 `stats-manager` 这种 module-owned shared stream manager 模式

### 2.6 Shell Boundary

容器 Shell 使用 module-private bidirectional websocket，会话和安全边界与统一 realtime topic gateway 不同。

因此：

- Shell websocket 不能作为 Runtime Events 的基础设施 authority
- Runtime Events 必须继续走统一 realtime topic pipeline

### 2.7 Existing Event-Related Code

当前仓库里“event”相关代码有三类：

1. `server/internal/realtime.Event`
   - transport-level realtime envelope
2. `server/internal/eventbus`
   - process-local domain event bus
3. `moduleapi.AuditEvent`
   - 审计领域事件

当前并不存在 live 的“container runtime events” canonical domain model。

结论：

- Runtime Events 不能复用 audit event model
- 也不应把 realtime transport-level `Event` 当成领域模型
- 需要在 container module 内新增自己的 canonical event domain

## 3. Authority 决策

### 3.1 Canonical Ownership

`Runtime Events` 的 canonical ownership 应落在：

- `server/modules/container/**`

原因：

- 当前产品能力是“容器管理”，不是“平台级任意 runtime event browser”
- 首个交付面是 container detail
- topic 命名、权限、history、filter 语义都属于 container domain，而不是平台 core realtime domain

### 3.2 Realtime Core 的职责

`server/internal/realtime/**` 只拥有以下 authority：

- topic subscription transport
- ticket issuance boundary
- websocket gateway
- in-memory fan-out hub

它不拥有：

- `RuntimeEvent` 领域定义
- event history policy
- provider normalization
- severity semantics
- container-specific filter semantics

### 3.3 Provider 的职责

provider adapter 只拥有：

- 连接具体 runtime 的 raw event source
- provider raw payload 解析
- provider-specific field 到 canonical field 的最小映射输入

provider 不拥有：

- canonical `RuntimeEvent` wire contract
- `severity` 决策
- `message` 文案
- `category`

### 3.4 Frontend Ownership

frontend 的 Events UX authority 应落在：

- `web/src/modules/container/**`

shared observability 只提供 stream primitive，不负责 container event domain。

## 4. Canonical Domain Model

## 4.1 Domain Layer Separation

本设计明确区分三层：

1. provider raw event
   - provider-owned
   - 不上抬到 OpenAPI / web contract
2. canonical `RuntimeEvent`
   - container module-owned domain fact
3. stream / history transport record
   - 包含 runtime context、sequence 等流式语义

### 4.2 RuntimeEvent

推荐 canonical domain fact：

```text
RuntimeEvent
  id
  resource_type
  resource_id
  event_type
  severity
  occurred_at
  attributes
```

字段约束：

- `id`
  - module-generated opaque id
  - 只要求在当前 history / replay window 内稳定唯一
- `resource_type`
  - 当前 phase 1 固定为 `container`
  - 保留未来 runtime-neutral 扩展空间
- `resource_id`
  - canonical container id
- `event_type`
  - stable canonical code
  - phase 1 推荐值：
    - `container.created`
    - `container.started`
    - `container.restarted`
    - `container.stopped`
    - `container.removed`
    - `container.oom_killed`
    - `container.health_status_changed`
    - `container.exec_started`
    - `container.exec_finished`
- `severity`
  - 由 canonical mapping 统一决定
  - provider 不得直接写入
- `occurred_at`
  - canonical event timestamp
- `attributes`
  - provider-normalized scalar metadata
  - 只允许文档化、稳定的 key
  - 不允许直接透传 provider 原始嵌套 payload

### 4.3 明确不进入 RuntimeEvent 的字段

以下字段本轮明确不进入 `RuntimeEvent`：

- `runtime`
  - 属于 stream context / subscription context，不属于 event fact
- `category`
  - 由 `event_type` 导出，避免双重真相
- `message`
  - UI 应基于 `event_type + attributes` 本地化
- provider raw action/status string
  - 除非已进入 canonical mapping，否则不应直出到 contract

### 4.4 RuntimeEventRecord

为 history / replay / websocket delivery，推荐独立 transport record：

```text
RuntimeEventRecord
  seq
  event
```

其中：

- `seq`
  - manager-assigned monotonically increasing sequence within one resource stream
  - 用于 replay ordering、去重和 UI stable key
- `event`
  - canonical `RuntimeEvent`

### 4.5 Runtime Context

`runtime` 不进入 `RuntimeEvent`，而是进入 stream context：

```text
RuntimeEventStreamContext
  runtime
```

phase 1 container detail 场景里：

- 一个 `resource_id` 只会对应一个当前 runtime context
- 若未来存在 Kubernetes / containerd layering，不让 `runtime` 与 event fact 混在同一层

### 4.6 Severity Mapping

`severity` 由 canonical mapping 决定，推荐保留三档：

- `info`
- `warning`
- `error`

推荐 phase 1 默认映射：

- `container.created` -> `info`
- `container.started` -> `info`
- `container.restarted` -> `warning`
- `container.stopped` -> `info`
- `container.removed` -> `info`
- `container.oom_killed` -> `error`
- `container.health_status_changed`
  - `healthy` -> `info`
  - `starting` -> `info`
  - `unhealthy` -> `warning`
- `container.exec_started` -> `info`
- `container.exec_finished`
  - 默认 `info`
  - 若后续补充 exit metadata，可按规则升级

该映射属于 container module authority，不属于 provider authority。

## 5. Runtime Abstraction 与 Provider Extensibility

### 5.1 不把 Events 直接塞进 Docker Provider

`Runtime Events` 不应作为“Docker provider 私有功能”直接上抬到 service / web。

如果这样做，会产生两个长期问题：

- Docker-specific event type 和字段外泄
- 第二个 runtime 接入时只能复制一套并行适配逻辑

### 5.2 推荐抽象：RuntimeEventSource

推荐在 container module 内新增 sibling abstraction：

```text
RuntimeEventSource
  Register / Subscribe / Close
  emit raw provider events
```

推荐治理：

- `Runtime` 继续承载请求式能力
  - list / detail / logs / shell / actions
- `StatsCollectorRuntime` 继续承载采样式 stats 扩展
- `RuntimeEventSource` 作为新的事件扩展接口，负责连续事件源

这比把 `Events()` 方法硬塞进现有 `Runtime` 更稳妥：

- 不要求所有 runtime 一次性实现事件能力
- 保持“可选扩展接口”与 `StatsCollectorRuntime` 一致
- 更容易支持未来多 source register

### 5.3 RuntimeEventManager

推荐引入 source-agnostic manager：

```text
RuntimeEventSource
  -> RuntimeEventManager
  -> Normalize
  -> History
  -> Realtime Topic
```

`RuntimeEventManager` 只知道：

- 已注册的 `RuntimeEventSource`
- canonical normalization rules
- history retention rules
- topic publish rules

它不应直接 import Docker-specific source 或写死 Docker 逻辑。

### 5.4 多 Runtime / 多 Source 未来路径

未来接入 Podman、containerd、Kubernetes 时：

- 只新增 source registration
- 不改上层 `RuntimeEvent` contract
- 不改 web page event rendering 语义

若未来 source 输出的是 `pod` 或 `workload` 级事件：

- 不能直接塞进 `container.events:{id}`
- 要么先归一化成“可归属到 container resource 的 container event”
- 要么转入未来 Kubernetes module-owned topic family

## 6. Backend 设计

## 6.1 API / Topic 位置

phase 1 推荐新增：

- HTTP history endpoint：
  - `GET /api/ops/containers/{id}/events`
- realtime topic：
  - `container.events:{id}`

topic ticket issuance 继续复用：

- `POST /realtime/subscriptions`

不新增第二套 websocket endpoint。

### 6.2 生命周期

phase 1 推荐 `RuntimeEventManager` 采用 Boot-started always-on 模式。

原因：

- 事件的价值在于“用户打开页前已经发生的事实”
- 如果像 logs 一样按订阅懒启动，会天然丢失订阅前事件
- 相比日志正文，生命周期事件频率更低，更适合 bounded in-memory retention

### 6.3 Subscription Model

推荐 phase 1 只做 detail-scope subscription：

- 一个 container detail 页面订阅一个 `container.events:{id}` topic

不在 phase 1 直接扩展：

- list-level `container.events.list`
- dashboard-level aggregate event feed

理由：

- 首个 UX authority 明确是 container detail
- 可以避免一开始就承担全局 fan-out 与聚合排序成本

### 6.4 History Model

phase 1 推荐：

- 不是 live-only
- 不是 persistent
- 采用 bounded in-memory replay

具体建议：

- manager 维护 per-container ring buffer
- history endpoint 返回 bounded recent records
- websocket 只负责 live append

不推荐 `live only`：

- 用户经常是在事件发生后才打开 detail
- 只看 live stream 会错过最关键的最近历史

不推荐 phase 1 持久化：

- 当前仓库没有 container runtime event durable store authority
- retention、index、search、cleanup、migration、权限与运维成本都会被提前放大

### 6.5 Buffer / Retention Policy

推荐 phase 1 bounded retention：

- per-container ring buffer：默认 `256`
- removed / inactive container history：TTL 驱逐
- 全局 manager：按 container id 做 LRU 或 lazy eviction

该策略目标是：

- 支持详情页 recent replay
- 避免 removed container 无限堆积
- 避免为了 phase 1 UX 引入数据库

### 6.6 Reconnect Backfill Strategy

phase 1 必须显式支持 reconnect 后补洞。

原因：

- 当前 `server/internal/realtime/hub.go` subscriber buffer 固定为 `8`
- publish 策略是 `drop oldest, keep latest`
- 当前 `web/src/shared/realtime/ws-client.ts` 负责 transport reconnect，但不负责 history backfill

因此 phase 1 最小正确策略应为：

- history seed 返回最近一段 bounded records，包含 `seq`
- live websocket 只 append 新 record
- reconnect 成功后，frontend manager 重新拉取 bounded history
- manager 以 `seq` 做 merge / dedupe，补齐断线窗口内缺口

不允许把“ws 能重连”误表述为“事件流天然不丢洞”。

### 6.7 Filtering

phase 1：

- history endpoint 返回 full bounded history
- UI 在已加载窗口内做 presentation filtering

phase 2：

- 后端再补 query filter：
  - `event_type[]`
  - `severity[]`
  - `before_seq`
  - `limit`
  - 可选 `from` / `to`

filter 语义 authority 仍属于 container module，不属于 provider。

### 6.8 History + Live Read Path

推荐 phase 1 读取路径：

```text
GET /api/ops/containers/{id}/events
  -> bounded history seed

POST /realtime/subscriptions { topic: container.events:{id} }
  -> GET /ws
  -> live append
```

原因：

- 与当前 stats “HTTP seed + websocket live” 模式一致
- 不要求 phase 1 扩展 core websocket handshake 去做 replay injection
- 避免在 realtime core 引入 container-specific history responsibility

### 6.9 Permissions

phase 1 推荐直接新增显式权限：

- `ops.container.events`

原因：

- Events 与 detail / logs / shell 是不同的可见面
- 后续若事件展示 exec、health、abnormal signal，权限边界应显式
- 当前 `server/modules/container/contract/permission.go` 尚无该权限；若现在不补，后续再拆会制造 topic、route 和 UI
  权限漂移

本设计不推荐 phase 1 临时复用 `ops.container.detail`。

若后续实现因收口成本不得不短期复用：

- 必须在 topic materials 中明确记录为过渡策略
- 必须声明 cleanup trigger
- 后续补独立权限时不得让 Docker-specific path 成为 authority

## 7. Frontend 设计

## 7.1 页面位置

phase 1 推荐在：

- `web/src/modules/container/pages/detail/index.vue`

新增 tab：

```text
Overview
Resources
Events
Logs
Shell
Health
Config
Network
Storage
Raw
```

核心决策：

- `Events` 放在 `Logs` 前面
- 保持 troubleshooting 顺序：Overview -> Events -> Logs -> Terminal
- 与当前 live detail owner 对齐：现有最后一个页签是 `raw`，不是新增第二套 `inspect` 命名

### 7.2 共享状态模型

推荐新增 module-owned `ContainerEventsManager`，位置保持在：

- `web/src/modules/container/shared/**`

职责：

- history seed
- websocket subscription lifecycle
- ring buffer append
- reconnect state exposure
- reconnect backfill merge
- selector 输出

页面本身只保留：

- filter
- search
- relative/absolute time toggle
- collapse state

realtime topic 常量 authority 继续放在当前容器模块 contract owner：

- `web/src/modules/container/contract/realtime.ts`

Events 不得新建平行 topic 常量文件去与现有 contract owner 竞争真相。

### 7.3 表现形态

phase 1 推荐“timeline-first，table-readable”的混合视图：

- 主视图为按时间倒序的 timeline list
- 每行保留 table-like metadata：
  - event label
  - severity
  - occurred_at
  - attributes summary

不推荐 phase 1 直接做纯表格：

- troubleshooting 更需要顺序感

也不推荐 phase 1 直接做复杂 grouped tree：

- 会过早引入折叠、聚合和跨组排序成本

### 7.4 Visual State

直接复用 shared visual stream state primitive：

- `idle`
- `connecting`
- `streaming`
- `paused`
- `reconnecting`
- `disconnected`
- `error`
- `empty`

推荐用 `StreamViewportStateSurface` 模式处理：

- first load skeleton
- empty history
- reconnect banner
- socket error

### 7.5 Severity Colors

推荐 phase 1：

- `info` -> neutral / primary
- `warning` -> warning
- `error` -> danger

颜色 authority 来自 canonical severity，而不是 provider raw status。

### 7.6 Phase 2 UX

phase 2 推荐补齐以下 UX：

- Event Type Filter
- Severity Filter
- Search
- Copy JSON
- Jump to Logs
- Timeline Collapse
- Relative / Absolute Time Toggle

`Jump to Logs` 语义建议为：

- 事件行点击后切到 `Logs`
- 以 event timestamp 作为日志窗口锚点
- 打开后展示事件前后 bounded log window

## 8. Streaming Strategy

### 8.1 是否复用现有 Streaming Pipeline

结论：

- live delivery：应复用现有 unified realtime topic pipeline
- bounded history：应新增 container module-owned history read path

也就是说：

- websocket transport 复用
- topic registry 复用
- reconnect strategy 复用
- visual state resolver 复用
- history ownership 不上提到 realtime core

### 8.2 Ordering Strategy

phase 1 ordering 建议：

- per-resource stream order 由 manager `seq` 保证
- `occurred_at` 用于展示时间
- UI 默认按 `seq desc` 渲染

不承诺：

- 多 source 全局 total ordering
- 跨 runtime 强一致全局排序

### 8.3 Reconnect Strategy

直接复用 `web/src/shared/realtime/ws-client.ts` 的现有策略：

- ticket issue retry
- websocket reconnect backoff
- non-retryable status fail-closed

但 Events 不能只停留在 transport reconnect。

phase 1 还必须补一层 module-owned recovery：

- socket reopen 后重新拉取 bounded history
- 以 `seq` merge / dedupe
- 避免 hub buffer 和断线窗口导致的事件缺口直接暴露给 UI

前端 manager 除了翻译 socket state，还要拥有 reconnect backfill 语义。

## 9. Compatibility Analysis

## 9.1 可直接复用

后端：

- `server/internal/realtime/TopicIssuerRegistry`
- `server/internal/realtime/RegisterSubscriptionRoutes`
- `server/internal/realtime/RegisterWebSocketGateway`
- `server/internal/realtime/Hub`
- container module `service` 作为 topic issuer owner 的现有模式

前端：

- `web/src/shared/realtime/ws-client.ts`
- `web/src/shared/observability/stream-viewport-state.ts`
- `web/src/shared/observability/log-ring-buffer.ts`
  - 当前实现是 generic，可直接承载 `RuntimeEventRecord`
- `web/src/modules/container/shared/stats-manager.ts`
  - 可复用其 refCount、idle grace、module-owned shared manager 模式

### 9.2 需要扩展

后端：

- `server/modules/container/runtime.go`
  - 新增 `RuntimeEventSource` sibling interface
- `server/modules/container/contract/topic.go`
  - 新增 `container.events:` topic prefix
- `server/modules/container/service.go`
  - 新增 events history / topic issuer owner
- container module boot lifecycle
  - 新增 `RuntimeEventManager`

前端：

- `web/src/modules/container/shared/**`
  - 新增 `events-manager`
- `web/src/modules/container/pages/detail/index.vue`
  - 新增 `Events` tab
- `web/src/modules/container/contract/realtime.ts`
  - 扩展 `container.events:` topic authority

### 9.3 应避免重复实现

- 不新增第二套 websocket gateway
- 不复用 shell private websocket 作为 events transport
- 不复制一份新的 ring buffer 实现
- 不新建第二套 container realtime topic 常量 owner
- 不在 provider 层直接生成 `severity`
- 不在 contract 中放 provider-authored `message`
- 不同时维护 `category` 和 `event_type`
- 不在页面局部维护一份长期 event store，与 module-owned manager 并行竞争 authority

## 10. Authority Analysis

| 能力 | Canonical Owner | 说明 |
| --- | --- | --- |
| `RuntimeEvent` | `server/modules/container/**` | container module-owned domain fact |
| raw provider event | runtime adapter package | 只在 provider 内部可见 |
| severity mapping | `server/modules/container/**` | canonical mapping，不属于 provider |
| history retention | `server/modules/container/**` | bounded ring buffer 与 TTL policy |
| websocket transport | `server/internal/realtime/**` | transport only |
| topic ownership | `server/modules/container/**` | `container.events:{id}` 由 container issuer 签发 |
| frontend stream manager | `web/src/modules/container/shared/**` | module-owned shared state |
| visual stream primitive | `web/src/shared/observability/**` | shared primitive，不拥有 container semantics |
| page-local filters | `web/src/modules/container/pages/detail/**` | presentation only |

## 12. Risks

### 12.1 Architecture Risks

- 把 Docker raw event type 直接暴露到 contract，会让第二个 runtime 接入时失控
- 把 `runtime` 放进 `RuntimeEvent` 本体，会在 Kubernetes / containerd layering 下产生歧义
- 把 `category` 和 `event_type` 双写，会制造 drift

### 12.2 Performance Risks

- Boot-started source 如果未来 runtime 数量增多，source fan-out 成本会上升
- per-container history 若没有 TTL / LRU，会对 removed containers 无限增长

### 12.3 WebSocket Fan-Out Risks

- 若未来很快扩成 list-level 或 dashboard-level event stream，topic fan-out 会增加
- phase 1 应先把 scope 限定在 detail-level

### 12.4 Ordering Risks

- provider 原始时间戳可能乱序
- 多 source 场景下无法承诺全局 total order
- phase 1 必须以 manager `seq` 作为 UI 排序 authority

### 12.5 Compatibility Risks

- Kubernetes 事件可能天然是 pod-level，而不是 container-level
- 未来若没有边界纪律，容易把 pod 事件硬塞进 container topic

### 12.6 UX Risks

- live-only 会错过关键最近历史
- 纯 timeline 不带 filter，会在 noisy container 上降低可用性

## 13. Final Recommendation

最终推荐：

- `Runtime Events` 归属 `server/modules/container/**`
- 以 `RuntimeEventSource -> RuntimeEventManager -> History -> Topic` 为 canonical backend pipeline
- `RuntimeEvent` 不携带 `runtime`、`category`、`message`
- `severity` 由 canonical mapping 决定，不由 provider 决定
- phase 1 使用 `HTTP history + realtime live` 的组合
- phase 1 只做 `container.events:{id}` detail topic
- frontend 通过 container module-owned `Events` manager 与 detail tab 消费
- bounded in-memory replay 是当前最稳妥的 phase 1 history strategy

这条路径最符合当前仓库既有模式：

- container module own domain
- realtime core own transport
- frontend module own feature state
- shared observability 只提供 primitive
