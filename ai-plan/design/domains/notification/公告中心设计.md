# 公告中心设计

本文档冻结 Announcement Center MVP 的领域边界、契约、数据模型和分阶段实施计划。后续实现公告中心时，`openapi/**`、`server/modules/announcement/**`、`web/src/modules/announcement/**`、权限菜单注册和 public recovery 主题必须以本文为当前 canonical authority。

公告中心是 `Graft` 的业务模块，不属于 `server/internal` core runtime。推荐后端归属为 `server/modules/announcement`，前端归属为 `web/src/modules/announcement`。

## 1. 公告与通知边界

Announcement 公告：

- 管理员主动发布。
- 面向长期阅读和平台信息传达。
- 典型内容包括系统维护、版本更新、安全提醒、平台通知。
- 支持草稿、发布、归档、置顶、过期和用户已读状态。
- 公告正文属于公告领域模型，不得写入 `notification_events`。

Notification 通知：

- 系统事件驱动。
- 面向个人消息和待处理事项。
- 通常由业务动作触发。
- 需要跳转到业务上下文或提示用户处理。

MVP 不把公告发布自动扇出为 Notification。理由：

- 公告是长期信息，通知是事件投递；直接联动会混淆收件箱语义。
- 公告列表自身可以承载未读状态和入口提醒。
- notification fan-out 需要额外治理投递目标、去重、冷却、跳转摘要和失败不阻断策略，不应压进公告 MVP。

预留点：

- 后续可以在公告发布成功后，通过 `server/internal/moduleapi.NotificationPublisher` 发布一条摘要通知。
- 摘要通知只能包含标题、级别、公告链接和必要上下文，正文仍以公告表为权威。
- 该联动必须作为独立 Phase 增加开关、幂等键和验证，不得把公告正文塞入 notification 事件事实。

## 2. 模块定位

后端模块：

```text
server/modules/announcement
```

前端模块：

```text
web/src/modules/announcement
```

模块负责：

- 公告 CRUD、发布、归档、软删除。
- 用户端可见公告列表、未读状态、标记已读、全部已读。
- 权限和菜单注册。
- OpenAPI 契约与 generated types 消费。
- 管理页和用户侧公告入口。

模块不负责：

- 系统事件判定。
- notification 投递 fan-out。
- 外部推送渠道。
- 告警、监控和值班升级。

## 3. 数据模型

MVP 使用两张表。

```text
announcements
  id
  title
  content
  level
  status
  delivery_mode
  pinned
  publish_at
  published_at
  published_by
  archived_at
  expire_at
  created_by
  updated_by
  deleted_by
  created_at
  updated_at
  deleted_at

announcement_reads
  id
  announcement_id
  user_id
  read_at
  created_at
```

字段语义：

- `level` 固定为 `info | warning | success | error`。
- `status` 固定为 `draft | published | archived`；MVP 允许 `draft -> published -> archived -> published`、`draft -> deleted`、`archived -> deleted`，不允许已发布公告直接删除。
- `delivery_mode` 固定为 `silent | popup`。`silent` 只进入公告中心，`popup` 对未读目标用户主动弹窗，同时仍进入公告中心。
- `pinned` 表示用户侧和管理侧是否优先展示。
- `publish_at` 表示公告生效时间 / 可见开始时间，参与用户端可见性判断；为空表示发布后立即生效，未来时间表示定时生效，过去时间用于补录。
- `published_at` 表示最近一次发布或重新发布动作实际发生时间；每次发布 / 重发布写入当前服务端时间，仅用于审计、后台展示、排序或记录，不参与用户端可见性判断。
- `published_by` 表示最近一次发布或重新发布公告的管理员用户 ID，与 `published_at` 配套；历史行不回填，直到下一次发布或重新发布才写入。
- `archived_at` 表示人工归档时间，仅在管理员手动归档时设置，重新发布时必须清空。
- `expire_at` 表示失效时间 / 可见截止时间，参与用户端可见性判断；为空表示长期有效，到期后用户端不可见但不自动归档。
- `created_by`、`updated_by`、`deleted_by` 记录管理操作人。
- `deleted_at` 使用新表统一软删除模型：`BIGINT NOT NULL DEFAULT 0`，`0` 表示未删除。

建模决策：

- 不需要独立 PostgreSQL schema，沿用模块表命名和当前 live migration 链。
- `announcements` 需要软删除，删除草稿是软删除，已发布公告建议先归档再删除或禁止直接删除。
- `announcement_reads` 不需要软删除，已读事实以唯一记录表达；归档后重新发布不清空已读记录。
- `announcement_reads` 必须对 `(announcement_id, user_id)` 建唯一约束，保证同一用户对同一公告只有一条已读记录。
- `announcement_reads.announcement_id` 可以 FK 到 `announcements.id`，用户 ID 按项目现有跨模块用户标识策略处理，避免跨模块 Ent edge 泄漏。
- `announcements` 列表查询需要围绕 `deleted_at`、`status`、`level`、`pinned`、`publish_at`、`expire_at` 和更新时间建立索引。
- 用户端查询固定包含 `deleted_at = 0`、`status = 'published'`、`(publish_at IS NULL OR publish_at <= now)`、`expire_at IS NULL OR expire_at > now`。
- `published_at`、`published_by` 不参与用户端可见性判断。
- archived 公告不在用户端继续显示；归档表示管理员主动下线长期信息，与 `expire_at` 到期隐藏不是同一件事。

枚举归属：

- 后端在 `server/modules/announcement/contract` 定义 typed string。
- OpenAPI schema 是跨端 wire contract authority。
- 前端在 `web/src/modules/announcement/contract` 消费 generated schema 或定义窄化 literal union，不在页面散落字符串。

数据库注释：

- 新 migration 必须包含中文 `COMMENT ON TABLE` 和所有新增列的中文 `COMMENT ON COLUMN`。
- enum、boolean、FK、JSON 字段注释必须说明业务语义和值域来源。
- live query 与 partial unique index 必须显式围绕 `deleted_at = 0`。

## 4. OpenAPI 与 API 计划

OpenAPI 是 HTTP 契约 authority。MVP 路径分管理端和用户端。

管理端：

```text
GET    /api/announcements
POST   /api/announcements
GET    /api/announcements/{id}
PUT    /api/announcements/{id}
POST   /api/announcements/{id}/publish
POST   /api/announcements/{id}/archive
DELETE /api/announcements/{id}
```

用户端：

```text
GET  /api/my/announcements
POST /api/my/announcements/{id}/read
POST /api/my/announcements/read-all
GET  /api/my/announcements/unread-count
```

`unread-count` 纳入 MVP，原因是 Phase 5 顶部入口需要轻量展示未读数，避免为了 badge 轮询整页列表。

列表过滤：

- 管理端支持 `status`、`level`、`pinned`、`keyword`、分页、排序。
- 用户端支持分页和可选 `unread_only`，排序默认 `pinned desc, publish_at desc, id desc`。

可见规则：

- 用户端只返回 `published`。
- `publish_at` 必须已到。
- `expire_at` 为空或未过期。
- `deleted_at = 0`。
- `archived` 不返回用户端。
- 已读状态通过当前 `user_id` 左连接 `announcement_reads` 计算。
- `delivery_mode` 不影响用户侧列表可见性，只影响前端是否对未读公告弹窗展示。

## 5. 权限与菜单

权限码：

```text
announcement.read
announcement.create
announcement.update
announcement.publish
announcement.delete
```

权限边界：

- 管理端接口需要公告管理权限。
- 用户侧查看公告只需要认证用户，不需要公告管理权限。
- `announcement.read` 表示管理端读权限，不表示用户端普通阅读能力。

菜单：

- MVP 不新增“系统管理”一级菜单。
- 公告管理先挂在现有服务/平台管理类菜单下，避免为了一个模块改变整体信息架构。
- 用户侧公告入口不进入左侧业务菜单，作为 shell global route 和顶部入口存在。

注册位置：

- 后端权限、菜单和路由归 `server/modules/announcement` 模块注册。
- 前端管理页路由归 `web/src/modules/announcement/bootstrap-routes.ts`。
- 用户侧 global route 也由 announcement 模块声明，壳层只装配。
- i18n key 归 `web/src/modules/announcement/locales/**`，shell 顶部入口如需文案则由模块或壳层明确归属，不能裸写中文或英文。

## 6. 后端实现计划

建议目录：

```text
server/modules/announcement/
  contract/
  migrations/
  store/
  module.go
  service.go
  route_registration.go
  route_handlers.go
  mapper_http.go
  module_registration.go
```

实现要点：

- `Register` 注册权限、菜单、路由和 migration。
- repository 封装列表、详情、创建、更新、发布、归档、软删除、已读、全部已读、未读数。
- service 负责校验、状态流转、用户可见规则和当前用户隔离。
- handler 只做 HTTP binding、auth context 提取和 DTO 映射。
- 发布行为把状态置为 `published`，写入 `published_at = now` 和 `published_by`；`publish_at` 仅保存请求传入的生效时间，未传时保存为空并表示立即生效；归档公告重新发布时清空 `archived_at`。
- 编辑已发布公告时，修改 `publish_at` 只代表修改生效时间，不更新 `published_at`。
- 归档行为把状态置为 `archived`，写入 `archived_at`，归档后用户端不可见；归档不修改 `published_at`、`publish_at` 或 `expire_at`。
- 删除草稿允许；删除已发布公告应要求先归档，避免误删仍在用户侧可见的信息。
- `read` 和 `read-all` 只影响当前 `user_id`。
- 时间口径：API 使用 RFC3339，业务时间入库使用 `timestamptz`，服务内部使用 `time.Now().UTC()`；软删除时间戳使用项目统一 epoch 口径。

必须验证的后端边界：

- 草稿不能出现在用户端。
- 未到 `publish_at` 的公告不能出现在用户端。
- `publish_at` 为空的已发布公告发布后立即出现在用户端。
- `expire_at` 过期后不能出现在用户端，但 `status` 仍保持 `published`，不自动写入 `archived_at`。
- `archived` 不能出现在用户端。
- 已读状态按 `user_id` 隔离。
- `read-all` 只影响当前用户。
- 权限不足时管理接口拒绝。

## 7. 前端实现计划

页面类型：`list-form-detail`。

管理端：

- 新增 `web/src/modules/announcement`。
- 新增 API 层消费 OpenAPI generated types。
- 新增公告管理列表页。
- 创建/编辑使用 Drawer 或 Dialog。
- 详情查看使用 Drawer。
- 发布、归档、删除作为行操作或详情操作。
- 状态、级别、展示策略、置顶、过期用 presenter/view model 统一映射。
- 正文按 Markdown 输入，管理详情、用户公告中心和 popup 弹窗统一使用 `web/src/shared/components/markdown/SafeMarkdown.vue` 渲染。
- `SafeMarkdown` 是唯一允许渲染公告正文 HTML 的边界，必须禁用 Markdown 原始 HTML，并在 `v-html` 前通过 DOMPurify 清洗。
- 创建/编辑表单只使用 textarea 和 Markdown 预览，不引入大体积编辑器。

用户侧：

- 新增 `/announcements` global route，不依赖左侧菜单。
- 顶部栏保留独立公告入口，职责是全局只读“公告中心”入口，显示未读 badge，不承载创建、编辑、发布、删除等管理能力。
- 顶部公告入口与 notification bell 需要保持明确间距和 tooltip/aria 语义，避免两个入口视觉混淆。
- `popup` 展示策略的未读公告由壳层挂载的公告 popup host 拉取并展示 TDesign Dialog；`silent` 不主动弹窗。
- popup 弹窗结构固定为级别 tag、未读 tag、标题、发布时间、Markdown 正文和“稍后查看 / 标记已读”操作。
- 同一条公告标记已读后不再弹出；“稍后查看”只在当前前端会话内暂不重复弹出。
- Phase 5 可在工作台提供最近公告摘要，但必须保持轻量，不把工作台变成模块列表聚合页。
- badge 使用 `/api/my/announcements/unread-count`。
- 管理端 create/update/publish/archive/delete 和用户端 read/read-all 成功后必须触发统一前端事件 `graft:announcement-changed`。
- 公告中心、顶部 badge 和 popup host 监听该事件并重新拉取，不在多个页面手写各自的刷新广播。

i18n 与 UI 约束：

- 全部可见文案 key-first。
- 页面内不散落状态字符串、级别字符串、权限码或路径魔法值。
- 复杂业务判断进入 presenter/composable/service，不堆在模板里。
- 可复用 notification 里的无业务共享组件，但不得复用会混淆通知语义的组件或文案。

## 9. 分阶段实施计划

Phase 0：探索与设计确认

- 目标：冻结边界、设计文档和 public recovery 主题。
- 范围：`ai-plan/design/domains/notification/公告中心设计.md`、`ai-plan/public/announcement-center-mvp/**`。
- 验证：文档审阅、`git diff --check`。
- 风险：设计与现有模块风格偏离。
- 提交：适合单独提交。

Phase 1：OpenAPI + 数据库迁移 + 后端领域模型

- 目标：建立 contract、migration、store/service 骨架和 module registration。
- 范围：`openapi/**`、`server/modules/announcement/**`、模块注册生成面。
- 验证：OpenAPI generate/check、SQL migration check、`go test ./modules/announcement/...`。
- 风险：migration version 冲突、generated artifact 漂移。
- 提交：适合单独提交。

Phase 2：后端管理端 API

- 目标：完成管理端 CRUD、发布、归档、删除和权限控制。
- 范围：`server/modules/announcement/**`、OpenAPI generated consumption。
- 验证：管理端 service/handler tests、backend focused tests。
- 风险：状态流转和删除边界不清。
- 提交：适合单独提交。

Phase 3：后端用户端 API + 已读状态

- 目标：完成 `/api/my/announcements`、已读、全部已读、未读数。
- 范围：`server/modules/announcement/**`。
- 验证：用户可见规则、用户隔离、read-all tests。
- 风险：用户身份提取和左连接统计错误。
- 提交：适合单独提交。

Phase 4：前端公告管理页

- 目标：完成管理列表、创建/编辑、详情、发布/归档/删除。
- 范围：`web/src/modules/announcement/**`、必要 route/i18n 聚合。
- 验证：`bun run lint:i18n`、模块测试、`bun run check`。
- 风险：表单校验与 generated types 对齐。
- 提交：适合单独提交。

Phase 5：用户侧公告入口 / 工作台摘要

- 目标：完成顶部公告入口、用户侧公告页、未读 badge、可选工作台最近公告摘要。
- 范围：`web/src/modules/announcement/**`、必要 shell/header/dashboard integration。
- 验证：用户侧交互测试、i18n governance、web check。
- 风险：与 notification bell 视觉和语义混淆，工作台变成大杂烩。
- 提交：适合单独提交。

Phase 6：测试、i18n、治理收尾、文档归档

- 目标：跑完整验证，更新 recovery trace，准备归档或 handoff。
- 范围：测试、docs、public topic。
- 验证：backend full、web full、migration、OpenAPI、i18n。
- 风险：既有 unrelated dirty files 干扰提交。
- 提交：适合单独提交。

## 10. 风险与待确认问题

- 公告管理菜单最终放在现有哪个一级菜单下，需要以当前 menu registry 真实结构为准。
- 顶部公告入口的视觉位置需要避免与 notification bell 竞争。
- 工作台摘要是否进入 MVP 取决于现有 dashboard 贡献机制；若没有轻量扩展点，应只做用户侧页面和顶部入口。
- `DELETE /api/announcements/{id}` 是否允许删除已归档公告，需要在 service 中明确规则。
- `publish_at` 缺省行为应统一为“立即生效”，发布动作时间由 `published_at` 单独记录。
- 后续 notification 联动必须独立设计，不属于 MVP 默认实现。
